# Maeterna — Web

React SPA for Maeterna's three portals: patient, doctor, and admin. Talks to the API in `apps/api` over HTTP; no server-side rendering.

## Stack

- **Routing**: TanStack Router, file-based (`src/routes/`) — route tree is auto-generated into `src/routeTree.gen.ts`, never edit it directly
- **Data**: TanStack Query + [`openapi-fetch`](https://openapi-ts.dev/openapi-fetch/), typed against a generated `src/lib/api.types.ts`
- **Auth**: [Better Auth](https://www.better-auth.com/) client (magic link, Google, passkey)
- **UI**: Tailwind CSS + shadcn/ui components, re-exported from `packages/ui` as `components/<name>`
- **Build**: [Vite+](https://viteplus.dev/) (`vp`)

## Structure

```
src/routes/patient/   Patient portal — mobile-first, max-w-lg, bottom nav
src/routes/doctor/    Doctor portal — desktop layout
src/routes/admin/     Admin portal — desktop layout
src/routes/signup/    Onboarding flow (role selection, profile completion, MBTT verification)
src/lib/              API client, auth client, session hook, thresholds, utils
src/mock/             Legacy mock data — see note below
src/components/       Shared app components (readings forms/lists, charts)
```

Each portal's top-level `route.tsx` is an auth guard: it checks the session on mount and redirects to that role's `/login` if missing or mismatched.

Patient-entered readings can be corrected or deleted from Reading history.
The dashboard remains read-only, and readings entered by a doctor do not expose
patient edit or delete controls. These mutations use the generated PATCH and
DELETE methods for `/patients/me/readings/{readingId}` and invalidate the shared
`["readings"]` query prefix.

## Running locally

Requires `apps/api` running (or a deployed API URL). Env vars go in `.env.local`:

```bash
VITE_API_URL=http://localhost:8787
VITE_BETTER_AUTH_URL=http://localhost:8787
VITE_APP_URL=http://localhost:3000
```

```bash
pnpm dev              # from repo root, runs this app via turbo
pnpm --filter ./apps/web dev   # or target this app directly
```

## API client

```ts
import { apiClient } from "@/lib/api-client"

apiClient.GET("/patients/me/readings")
apiClient.POST("/profile/complete", { body })
```

`src/lib/api.types.ts` is generated by `openapi-typescript` and shouldn't be hand-edited beyond small, targeted fixes. **Regenerate it from a running server's `/openapi.json`** (e.g. `npx openapi-typescript http://localhost:8787/openapi.json -o src/lib/api.types.ts`), not from `apps/api/maeterna-openapi.yaml` — that file has drifted from the real routes and will produce a large, unrelated diff if used as the source.

## Note on `src/mock/`

`src/mock/db.ts` and `src/mock/auth.ts` are leftovers from an earlier prototype that used in-memory mock data and a fake sessionStorage-based session. The app has since moved to the real API and Better Auth — these files are largely dead code now, kept alive only because a few components still `import type` domain types (`Reading`, `ReadingType`, etc.) from `mock/db.ts`. Don't add new logic to them; use `apiClient` for data and `authClient`/`useSession()` (`src/lib/auth-client.ts`, `src/lib/session.ts`) for auth.
